## Findings

### F1 — Generator emits `example-value` placeholder in command `Example:` strings, which breaks live happy_path dogfood tests on strict-validation APIs (Default gap)

- **What happened:** Every absorbed endpoint command's `Example:` string was emitted with `example-value` as the literal placeholder for required path/query params (e.g., `coin facts-grade --pcgs-no example-value --grade 42 --plus true`). Dogfood's live happy_path test passes the example verbatim. PCGS rejects malformed inputs with `IsValidRequest: false`, surfacing as a typed exit-3 failure. 8 of the initial 10 dogfood failures had this root cause.
- **Scorer correct?** Mixed. The scorer's happy_path expectation (exit 0 on the example) is correct in principle. The generator's `example-value` defeats it. Fix-side: generator.
- **Root cause:** `internal/generator/` templates emit `Example: "  <cli> <cmd> --<flag> example-value"` for every required string-typed param. The placeholder propagates straight into the live test.
- **Cross-API check:**
  - **PCGS** — confirmed 8 failures in this run.
  - **Stripe** — `customers/{id}`, `subscriptions/{id}` etc. would reject `example-value` against the `cus_*`/`sub_*` regex; same shape of failure.
  - **GitHub** — `repos/{owner}/{repo}` rejects `example-value` (404 from `GET /repos/example-value/example-value`).
- **Frequency:** every CLI with required path/query params on at least one command (~90% of catalog).
- **Fallback if the Printing Press doesn't fix it:** the agent has to hand-edit each command's `Example:` after generate. Catches happen at Phase 5 live dogfood, not Phase 4 shipcheck (because shipcheck's `validate-narrative --full-examples` runs offline with `--dry-run` which never sees the bad value).
- **Worth a Printing Press fix?** Yes — surfaces only at the latest, most expensive phase otherwise.
- **Inherent or fixable:** fixable. Three reinforcing routes:
  - **(a) Research-driven:** read `research.json` `narrative.quickstart` and `narrative.recipes`; back-propagate the first known-good value per param-name to the matching command's `Example:` field. Falls back to `example-value` only when no quickstart/recipe references the command.
  - **(b) Spec-driven:** consume optional `example:` field on each param in internal YAML (and `example`/`x-example` in OpenAPI); when set, use that value in the `Example:` string. Adds nothing to the parser surface — it just lifts an existing field through.
  - **(c) Both:** prefer (a) when available, fall back to (b), fall back to `example-value`.
- **Durable fix:** (c) — research-driven with spec-driven backup. Drives example quality from the agent's own work (research.json) without requiring spec authoring discipline.
- **Test:** generate a sample CLI for an API with known-good examples in `narrative.quickstart`; verify the generated `Example:` strings use those values. Negative: generate for an API with no quickstart entries; verify falls back to `example-value`.
- **Evidence:** [PCGS shipcheck proof](manuscripts), 8 failures all tagged `[happy_path]` with `--pcgs-no example-value --grade 42 --plus true` or equivalent.
- **Related prior retros:** None (no prior retros exist).

### F2 — Dogfood's `error_path` probe hardcodes `__printing_press_invalid__` and doesn't allow commands to opt out or declare alternate semantics (Scorer bug)

- **What happened:** PCGS's images endpoints (`/coindetail/GetImagesByCertNo`, `/banknotedetail/GetBanknoteImagesByCertNo`) return `IsValidRequest: true, ServerMessage: "Request successful"` for ANY well-formed cert string — including `__printing_press_invalid__`. The CLI faithfully exits 0. Dogfood's error_path test expects exit non-zero. Two failures persisted across multiple fix loops until we added a CLI-side heuristic.
- **Scorer correct?** Partially. error_path is correctly looking for "CLI should reject bad input." But the probe assumes the API validates strictly; when the API doesn't, no CLI fix is possible without adding API-specific heuristics, which is a deeper invariant to ask of every endpoint.
- **Root cause:** `printing-press dogfood` (`internal/dogfood/` or equivalent CLI subcommand) substitutes a hardcoded bogus value for the first positional arg on every command, regardless of the command's annotations or the API's response-shape behavior.
- **Cross-API check:**
  - **PCGS** — confirmed (images endpoints).
  - **Wikidata / NOAA / similar lookup-by-ID APIs** — typically return an empty struct or `{}` rather than 404 for unknown IDs.
  - **Stripe `customers.list` with arbitrary filter values** — returns 200 with empty `data: []`; same shape of "no validation at this layer."
- **Frequency:** subclass: APIs that don't validate input strictly at the lookup endpoint. Probably 10-15% of catalog (the long tail of REST APIs with lenient/echo-style responses).
- **Fallback:** each affected CLI needs either a heuristic (like our `classifyPCGSEnvelope` images branch) or to opt out of error_path.
- **Worth a Printing Press fix?** Yes — the error_path failure is otherwise a permanent yellow on the dogfood report for these APIs.
- **Inherent or fixable:** fixable. Two reinforcing routes:
  - **(a)** Allow a command annotation `pp:no-error-path-probe = "true"` (or `error_path = "soft"`) that tells dogfood the API's response for bad input is structurally indistinguishable from success and the test should be skipped.
  - **(b)** Detect the case at runtime: when error_path returns exit 0 AND the response includes `IsValidRequest: true` (or vendor-specific success markers from a per-API hint file), classify as soft-pass with a warning rather than a hard fail.
- **Durable fix:** (a) annotation-driven. Cheaper; explicit; auditable per-command.
- **Test:** flag a command with `pp:no-error-path-probe = "true"` and verify dogfood skips error_path for that command but still runs help / happy_path / json_fidelity.
- **Evidence:** PCGS images endpoints; two failures that persisted through the example-value fix and were only resolved by adding a CLI-side heuristic.
- **Related prior retros:** None.

### F3 — `Highlights` / `Long:` / README "Unique Features" prose are emitted from `novel_features` (planned) at generate-time, not from `novel_features_built` (shipped); drift requires hand-patching even after dogfood resync (Template gap)

- **What happened:** `research.json`'s `novel_features` array was the source of truth for: (a) `internal/cli/root.go` `Long:` field's "Highlights (not in the official API docs)" block, (b) `README.md` `## Unique Features`, (c) `SKILL.md` `## Unique Capabilities`. When the agent's manifest collapsed during Phase 1.5 (`quota` → stderr+flag, `stale` → `refresh --dry-run`, `coin ingest` → `coin batch --list-certs`, `coin pop1` → `search --max-pop`), the generator had already emitted these prose blocks referencing the original 11 features. Dogfood's auto-sync rewrote `novel_features_built` and the standalone README/SKILL highlight blocks, but the `Long:` text inside root.go and the prose body that wraps the highlight blocks (README intro paragraphs, surrounding examples) still referenced the dropped commands. Hand-patching needed across 4 files.
- **Scorer correct?** N/A — verify-skill's mechanical match did catch the dead-command references in the Long: text (`pcgs-pp-cli quota` / `pcgs-pp-cli stale` etc.), forcing the hand-edit. The scorer worked; the generator emitted the problem.
- **Root cause:** generator templates for `root.go` `Long:`, `README.md` body prose, and `SKILL.md` body prose substitute from `research.json` at generate-time. Dogfood's later resync (`pp:novel_features_built`) only rewrites the explicit Highlight/Unique-Features blocks, not the surrounding prose or the root command's `Long:` field.
- **Cross-API check:**
  - **PCGS** — confirmed (4 dropped novel features still appeared in root.go Long: text after build).
  - **Any combo CLI where the agent shrinks scope at Phase 1.5** — same pattern.
  - **Any run where Phase 3 build can't realize an absorbed feature and downgrades it to a stub** — Highlights references the planned-not-built feature.
- **Frequency:** subclass: runs where the surface evolves between research authoring and build completion. Most non-trivial CLIs.
- **Fallback:** agent hand-patches root.go + README + SKILL prose after dogfood resync. Easy to miss when the strings are scattered across multiple files.
- **Worth a Printing Press fix?** Yes.
- **Inherent or fixable:** fixable.
- **Durable fix:** Two-part. (a) Make all prose-substitution sites read from a single source-of-truth (preferably a post-build `novel_features_built` array re-emitted into research.json by dogfood); the generator templates take `novel_features_built` when present, fall back to `novel_features` only on the first generation pass. (b) Expand dogfood's resync scope to ALSO rewrite root.go's `Long:` field and the surrounding README/SKILL prose paragraphs that name specific commands.
- **Test:** Generate a CLI with a 5-feature manifest. Run dogfood. Verify only the actually-built commands appear in root.go Long:, README.md Unique Features, and SKILL.md Unique Capabilities. Negative: verify dropped features do NOT appear anywhere in the user-facing prose surface.
- **Evidence:** PCGS shipcheck-fix commit, README.md and SKILL.md sed-style patch removing references to `quota`/`stale`/`coin pop1`/`coin ingest`.
- **Related prior retros:** None.

### F4 — `classifyAPIError` template in generated `helpers.go` re-classifies pre-typed `*cliError` instances via string-matching on the error message, clobbering custom typed exit codes (Template gap)

- **What happened:** When I added envelope classification in `resolveRead` returning `usageErr(exit 2)` for `IsValidRequest: false` and `notFoundErr(exit 3)` for "No data found", every absorbed command's `if err != nil { return classifyAPIError(err, flags) }` was intercepting the typed error and re-mapping it to `apiErr(exit 5)` because `classifyAPIError`'s switch hit `default: return apiErr(err)` for our message text. Smoke test showed exit code 5 instead of 2. Required a one-line guard in the helper to pass through `*cliError` instances.
- **Scorer correct?** N/A (no scorer involved).
- **Root cause:** generator template `command_promoted.go.tmpl` (or equivalent) emits `if err != nil { return classifyAPIError(err, flags) }` for every absorbed command. `helpers.go` template emits a `classifyAPIError` that does not test for pre-typed errors.
- **Cross-API check:**
  - **PCGS** — confirmed (had to patch helpers.go to pass-through *cliError).
  - **Any API with envelope-style soft errors** (Salesforce SOAP-style envelopes, PayPal v2 with success:false, NUMista with API error subobjects, OAuth2 metadata endpoints that mix structural success with semantic failure).
  - **APIs with mixed success/failure responses** where the CLI maintainer wants typed exit codes that don't map directly to HTTP status codes (any rate-limited API where the limit is signaled in-body, any vendor that returns 200 + `errors: [...]`).
- **Frequency:** every CLI that adds custom typed errors. Could be 30-40% of catalog as agents adopt envelope-classification patterns.
- **Fallback:** agent has to patch `classifyAPIError` after every generation. Easy to forget; failure mode is silent (wrong exit code, harder to spot than a crash).
- **Worth a Printing Press fix?** Yes — one-line template fix.
- **Inherent or fixable:** fixable. Single-line guard at the top of `classifyAPIError`.
- **Durable fix:** Generator template `helpers.go.tmpl` (or equivalent) `classifyAPIError` body should start with:
  ```go
  var typed *cliError
  if errors.As(err, &typed) {
      return err
  }
  ```
  before the string-matching switch. This makes the function strictly enrich, never overwrite.
- **Test:** Hand-author a usageErr in a printed CLI's command; verify `classifyAPIError` returns it unchanged. Negative: verify HTTP-error mapping (401 → authErr) still works as today.
- **Evidence:** PCGS commit "phase5: typed exit codes (envelope classifier in resolveRead) + real example values" — the helpers.go portion of that diff is the one-line guard described above.
- **Related prior retros:** None.

## Anti-patterns
- Emitting placeholder strings (`example-value`) into runtime-tested artifact slots is a pattern with hidden cost — it works for offline tests, fails for live tests, and surfaces at the most expensive phase.
- Substituting from "planned" data sources at template-time without a regeneration path for "actual" data sources accumulates drift any time the manifest evolves during build.
- String-matching error classification that runs AFTER a pre-typed error has already been authored is fragile by construction; it should be additive only.

## What the Printing Press Got Right
- The MCP cobratree mirror (runtime, not generate-time) is exactly the right architecture for this kind of CLI. The 6 novel commands and 12 absorbed endpoints all surface as MCP tools without any per-command MCP boilerplate.
- The `cliutil.AdaptiveLimiter` and the cache layer in `client.do()` made the LogHook injection a one-line plumbing change — the request lifecycle was already cleanly separated.
- `printing-press dogfood --json` and the `phase5-acceptance.json` schema are the right machine-readable shape; iterating against them across 4 fix loops was fast and unambiguous.
- The internal-YAML spec format made the spec-source-of-truth question trivial after the user clarified the Numista-vs-PCGS mismatch — switching from "the file named pcgs-swagger.yaml" to "an authoritative spec from the right vendor" took one file edit.
- The validate-narrative leg of shipcheck (with `--full-examples`) was the right catch for the `coin batch --file <path>` examples that referenced nonexistent fixture paths; we shipped a real `examples-pcgs-coin-list.csv` in the CLI dir to satisfy it AND give users a working starter file.
- The lock heartbeat (`lock acquire` / `lock update` / `lock promote` / `lock release`) is the right shape — particularly the `lock promote`'s strict acceptance-gate enforcement that caught the 6 failed tests and forced the user-CLI conversation that produced the input-normalization fix.
- The Phase 1.5 absorb gate's adversarial-cut pass produced a customer model and 6 killed candidates with one-sentence reasons — the audit trail let us defend the slimmer surface to the user.
